# Code signing

By default the release workflow produces **unsigned** installers. On first launch
users see Windows SmartScreen ("Windows protected your PC" → *More info* → *Run
anyway*) or macOS Gatekeeper (right-click → *Open*). Signing changes that.

## What the Corner Spore self-signed cert does (and doesn't)

Switchboard's Windows builds can be signed with the shared **"The Corner Spore
Code Signing"** certificate (a self-signed two-tier chain: *Corner Spore Root CA*
→ *Code Signing* leaf, generated by `mcp-command-center/scripts/make-cert.ps1`).

- ✅ Produces a **verified "The Corner Spore" publisher signature**.
- ✅ Trusted with **no warning** on any machine where the Root CA is installed
  (your own / org machines — locally or via Group Policy / Intune).
- ❌ Does **not** clear SmartScreen for the **public** — that needs a CA-issued OV/EV
  cert with reputation.
- ❌ Does **nothing for macOS Gatekeeper** — that needs an Apple Developer ID.

In short: great for machines you control, not a public-trust fix.

## Sign locally (Windows)

```powershell
# point electron-builder at the cert + its password, then build
$env:CSC_LINK = (Resolve-Path .\certs\tcs-code-signing.pfx)
$env:CSC_KEY_PASSWORD = '<pfx password>'
npm run dist:win
```

`dist/*.exe` will be Authenticode-signed by "The Corner Spore Code Signing".
If `CSC_LINK` is unset, `npm run dist:win` still works and yields unsigned builds.

## Trust the signature on another machine you control

```powershell
# as Administrator
Import-Certificate -FilePath certs\tcs-root-ca.cer -CertStoreLocation Cert:\LocalMachine\Root
```

After that, signed installers show "The Corner Spore" as a verified publisher with
no Authenticode warning on that machine.

## Sign in CI

Add repository secrets (**Settings → Secrets and variables → Actions**):

- `CSC_LINK` — base64 of the `.pfx` (the `certs/tcs-code-signing.pfx.b64` contents),
  or an https URL to the cert
- `CSC_KEY_PASSWORD` — the `.pfx` password

Then uncomment the `env:` block on the *Build ${{ matrix.platform }} installer* step
in `.github/workflows/release.yml`:

```yaml
        env:
          CSC_LINK: ${{ secrets.CSC_LINK }}
          CSC_KEY_PASSWORD: ${{ secrets.CSC_KEY_PASSWORD }}
```

> Only add that block once the secrets exist — an empty `CSC_LINK` makes
> electron-builder treat `""` as a cert path and fails the build. With no block
> (the default) builds are unsigned and always succeed.

## macOS (Developer ID + notarization)

Requires a paid Apple Developer account. Export your *Developer ID Application*
cert as a `.p12`, then add `CSC_LINK`/`CSC_KEY_PASSWORD` (base64 of the p12 + its
password) plus `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD`, `APPLE_TEAM_ID` as
secrets. electron-builder signs and notarizes automatically when present.

> ⚠️ `certs/`, `*.pfx`, and `*.pfx.b64` are **gitignored** — never commit the
> private key.
